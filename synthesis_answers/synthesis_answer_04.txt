Audit question: How are changes such as maintenance releases, configuration changes, or software updates evaluated to determine if design controls apply?
Chunks used: 37
================================================================================
The excerpts provide various insights into how changes such as maintenance releases, configuration changes, or software updates are evaluated in relation to design controls for the Halcyon 4.0 system.

1. **Process for Evaluating Changes:**
   - Necessary design changes are managed via a change request, with regression tests and risk assessments performed for each update. This process is documented in the product development process DND01 Design and Development rev03 and DND01F02 Maintenance Release Document rev01 [Regulatory / Test Reports - Regulatory Compliance / D78333 Ethos 4.0 Description of interoperability Study Rev01.docx.pdf, Page 21].
   - For software maintenance planning, configuration is managed per specific software configuration management and design and development processes. Changes such as upgrades or bug fixes are evaluated and implemented following these procedures [Regulatory / D70036 HAL 4.0 Software Description Report China Submission Rev 03 .docx.pdf, Page 27-28].
   - Maintenance releases like HAL 4.0 MR1 undergo risk analysis to evaluate the impact of changes on system hazards and risks. No new hazards were identified for MR1, and no existing risks changed due to the maintenance release [Design Changes / Maintenance Releases / D69533 HAL 4.0 MR1 Maintenance Release Document Rev09.docx.pdf, Page 27].

2. **Design Control Application:**
   - Design controls for maintenance releases are defined and the scope of changes is controlled through formal documentation and release processes. For HAL 4.0 MR2, no new functionality was introduced but the impact on the software build and product version was evaluated as it was a commercialization release for a new model in China [Design Changes / Maintenance Releases / D95424 HAL 4.0 MR2 (HAL Evolve) Maintenance Release Document Rev 04.pdf, Page 9].
   - Configuration parameters and their impact are verified to ensure that updates do not occur in clinical mode, which indicates checks are in place to maintain system integrity and confirms adherence to design controls [Design Output Specification / Design Trace Matrices / D68599 Hal 4.0 Design Verification Trace Matrix Rev 02.pdf, Page 485].

3. **Concrete Determinations for Halcyon 4.0:**
   - HAL 4.0 MR1 and its risk assessment determined that none of the findings warranted additional hazards or changes in risk, indicating a structured risk evaluation process tied to design controls [Design Changes / Maintenance Releases / D69533 HAL 4.0 MR1 Maintenance Release Document Rev09.docx.pdf, Page 27].
   - HAL 4.0 MR2 changes did not lead to new features or functions, emphasizing a management process for alterations without significant impacts, showing design control consistency [Design Changes / Maintenance Releases / D95424 HAL 4.0 MR2 (HAL Evolve) Maintenance Release Document Rev 04.pdf, Page 9].

In summary, changes to Halcyon 4.0, such as maintenance releases, are evaluated through well-documented procedures involving change requests, risk assessments, and design controls. Documentation processes support the tracking and justification of these changes to ensure compliance with design control requirements.
